# Backend Requirements

## Overview
Node.js and Express.js REST API server. Connected to Supabase as the database. Drizzle is used to define table schemas and generate database migrations. Emails are sent via Nodemailer using a Gmail account with App Passwords. AI follow-up questions are generated using the Gemini API (use gemini-2.5-flash-lite).

---

## Database Tables (defined via Drizzle)

### users
- id
- name
- email (unique)
- password (stored as a secure hash)
- role (either "user" or "admin")
- otp (the current one-time password sent to the user)
- otp_expiry (timestamp when the OTP becomes invalid)
- is_verified (whether the user has completed OTP verification)
- created_at

### complaints
- id
- user_id (references users)
- complaint_text (the original complaint written by the user)
- ai_question (the follow-up question generated by AI)
- user_answer (the user's answer to the follow-up ai-generated question)
- created_at

## API Endpoints

### Auth

#### POST /api/auth/send-otp
- Accepts: name, email
- Creates a new unverified user record
- Generates a 6-digit OTP, saves it with an expiry time, and sends it to the user's email
- Returns: success message

#### POST /api/auth/register
- Accepts: email, otp, password
- Checks that the OTP matches and has not expired
- Marks the user as verified
- Saves the password as a secure hash using bcrypt
- Returns: success message

#### POST /api/auth/login
- Accepts: email, password
- Checks that the user exists, is verified, and the password matches
- Sets the JWT as a cookie in the response
- Returns: the name, email, user's role

#### POST /api/auth/logout
- Clears the JWT cookie
- Returns: success message

#### GET /api/auth/me
- Reads the JWT from the cookie and returns the logged-in user's name, email, and role
- Used by the frontend on page load to verify session and determine which pages to show

---

### Complaints

#### POST /api/ai/question
- Accepts: complaint_text
- Sends the complaint text to Gemini and asks it to generate one relevant follow-up question
- Returns: the AI-generated question

#### POST /api/complaints
- Accepts: complaint_text, ai_question, user_answer
- Creates a new complaint record linked to the logged-in user
- Returns: the created complaint

#### GET /api/complaints/my
- Returns all complaints submitted by the currently logged-in user

#### GET /api/admin/complaints
- Only accessible to users with the admin role
- Returns all complaints from all users, including the name and email of the submitting user

## Email Service
- Uses Nodemailer configured with a Gmail account
- Gmail App Password is used instead of the main account password
- Sends OTP emails during registration with a simple plain-text body containing the code

## AI Service
- Uses the Gemini API
- Model: gemini-2.5-flash-lite (lightest model available on the free tier as of May 2026)
- Called when a user submits their complaint text to get a follow-up question
- The prompt instructs Gemini to return exactly one short follow-up question relevant to the complaint

## CORS
- The backend must allow requests from the frontend origin (the local port the frontend runs on)
- Credentials must be allowed so cookies are sent with cross-origin requests

## Server
- The backend runs on a defined local port (e.g. 3000)

## Environment Variables Required
- Supabase database connection URL
- Gmail sender email address
- Gmail App Password
- Gemini API key
- JWT secret
- Backend port

## Notes
- Every newly registered user is assigned the "user" role by default
- Promoting a user to admin can only be done by directly changing the role field in the database — there is no API for this
- OTP expires after 10 minutes
- Sessions are managed using JWT tokens — the token is issued on login and stored in a cookie
- The cookie must not be HttpOnly, not Secure, and not SameSite Strict (kept this way for easier local testing)
- Admin routes additionally check that the user's role is admin
- If a user tries to register with an email that already exists, return a clear error message
